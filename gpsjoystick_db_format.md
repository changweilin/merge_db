# GPS Joystick `.db` 合併與開發指南 (Merge & Development Guide)

如果您要開發一個「合併多個 .db 檔案」的應用程式，本指南將協助您在不使用官方 SDK 的情況下，透過二進位修改實現檔案合併。

---

## 1. 合併核心策略：模板替換法 (Template In-place Merge)

由於 Realm 資料庫的結構極為嚴密，手動拼接檔案會導致所有記憶體偏移值失效。因此，「合併」的最穩健實作方式是 **「選取大模板空間，覆寫新資料內容」**。

### 步驟 A：選取「容器」模板 (Selecting the Host)
1. 讀取所有欲合併的 `.db` 檔案，計算每個檔案的 **Run 容量**：
   * `檔案容量 = (該檔案 Float64 葉節點數量 / 2) * 1000` (經緯度各佔一半)。
2. 挑選 **容量最大** 的那個檔案作為「主容器」。

### 步驟 B：重新排版內容內容
1. 從各個檔案中依序提取出所有軌跡名稱 (Names) 與座標串 (Tracks)。
2. 計算「總需求點數」：所有軌跡的座標總和。
3. **安全檢查**：確保 `總需求點數 <= 主容器總容量`。若超出，則必須捨棄部分軌跡。

## 2. 合併實作技術細節

### 2.1 修改路線名稱 (Names Injection)
這是在合併過程中讓 GPS Joystick 能顯露出不同路線的關鍵。

* **挑戰**：名稱節點 (`AAAA + 0x11`) 是定長的。如果你要合併的軌跡名稱比原模板多，直接寫入會破壞結構。
* **開發規則**：
  1. 走訪主容器檔案，找到所有原始的路線名稱位址。
  2. 將你要合併的新路線名稱，依照順序覆寫進去。
  3. **關鍵！**：若新名稱較短，剩餘部分請填入 `0x00` (Null byte) 或空白。若新名稱較長，必須截斷成與原模板節點一致的長度。
  4. 如果原容器只有 10 個名稱位：則合併後的檔案最多也只能顯示 10 條名稱。

### 2.2 座標原地覆寫 (Coordinates Overwrite)
這是合併的主體，需保持座標的「連續性」。

* **寫入規則**：
  1. 準備一個巨大的連續 Float64 陣列，把檔案 A、檔案 B、檔案 C 的座標按順序全部排進去。
  2. 從主容器的 `latRun[0]` 與 `lonRun[0]` 開始，依序將這串龐大的座標覆寫進去。
  3. 不同路線之間**不需要**在中間留空（除非你想手動控制 B-Tree 索引，這非常困難）。

### 2.3 跨路線跳轉處置 (Segmentation Gap)
因為我們在步驟 2.2 是將所有座標強行連續排在一起，GPS Joystick 在讀取時會把這幾萬個點看成「一條超級長路徑」。

* **開發解決方案**：
  在相鄰兩條路線的「交接處」，刻意在二進位中寫入一組**「極大位移座標」**。
  * 例如：在檔案 A 的最後一點與檔案 B 的第一點之間，確保距離超過 300 公里。
  * 這樣 GPS Joystick 的 UI 就會自動因為距離跳轉太大，將其視為獨立的不同路徑片段（對應步驟 2.1 修改的各個名稱）。

## 3. 合併開發者的邊界條件 (Edge Cases)

| 狀況 | 開發者應對方案 |
|-----|---------------|
| **空間不足** | 偵測到總點數超過模板容量時，應主動提示使用者「目標資料庫空間不足，僅能合併前 N 條」或「請先在 App 內建立一條空的長路徑以擴張資料庫空間」。 |
| **高度與速度 (Run 2)** | 如果您也想保留高度資料，請同步覆寫 `Run 2` 的節點。若不在意，可將其全部填入 `0x00` 或維持原樣。 |
| **名稱數量不足** | 若原檔案只有 5 個名稱位，但你要合併 10 條路線。你必須告知使用者多出的 5 條將會被併入之前的路線名稱中（透過 300km 跳轉法區分）。 |
| **Little-Endian 警示** | 所有的數值 (Float64, Uint32) 寫入時都必須確保使用 Little-Endian 格式。 |

## 4. 驗證工具開發建議

開發合併 App 時，建議內建一個簡易的二進位校驗器，檢查合併後的檔案：
1. 檔案大小是否與原模板完全一致 (Byte-for-byte Size parity)。
2. `0xAAAAAAAA` 識別碼的總數是否發生變動 (發生變動代表你寫入時發生位移錯位，檔案會損毀)。
3. 使用 `isFinite()` 檢查寫入後的浮點數是否正確，避免產生 `NaN` 座標導致程式閃退。
