## 根本原因分析

### 問題一：bpe 污染（影響 2026 檔段 [0]–[15]）

```
路線 ID 欄（type=0x43）← 假 AAAA marker 截斷前一個節點
                        ← 路線 ID 值 0x41414141... 被誤讀為 AAAA marker
                        ← 受影響節點 bpe=0，以 fallback bpe=1 補救
                        ← 部分 ID 值讀錯 → 同一 ID 散布全資料集
```

### 問題二：idFirst..idLast 範圍覆蓋（已修復）

```
舊行為：segment = idFirst[id]..idLast[id]
        → 若 id 出現在位置 [0] 和 [67000]，範圍覆蓋 0..67000（全部資料）
        → TSP 計算 67001 個點的環形路徑 → 洲際距離

新行為：segment = 所有 ids[i] == id 的精確位置清單
        → 只包含真正屬於這條路線的點
        → 仍有跳躍 → maxConsecutiveJumpKm 篩選
```

### 修復效果

| | 修復前 | 修復後 |
|-|--------|--------|
| 2026 大距離路線 | 出現 51,687 km 等距離 | 精確過濾 + 跳躍 skip |
| 2026 段 [8] 點數 | 66,794 點（幾乎全部） | 1,269 點（精確） |
| 2020 全部路線 | 正常 | 正常（無變化） |

## 建議

1. **段 [0]（2020 檔，2840 點）**：目前因超過 1024 點上限被略過。若確認是正常長時間錄製路線，可考慮提高 skipLargeThreshold 或單獨處理。

2. **2026 檔段 [0]–[15] 16 條污染路線**：精確 ID 值已被跳躍過濾略過。由於 bpe 污染無法從現有資料中恢復正確 ID 對映，這 16 條路線的 TSP 優化無法執行，建議在匯出時保留原始座標順序。

3. **空 UUID（2020×1, 2026×2）**：為資料完整性問題，不影響現有路線的 TSP 處理。可在 UI 加入提示告知使用者有空白路線存在。

4. **2026 檔段 [8] 的 1,269 個點**：雖跨越全球（經過布達佩斯、日本等地），在 GPS Joystick 的使用情境下可能是合法的「全球示範路線」。若使用者確認為有效路線，可將 maxConsecutiveJumpKm 設為 0 關閉過濾，單獨執行。
